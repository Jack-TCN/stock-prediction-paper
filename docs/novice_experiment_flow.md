# 论文小白版实验流程说明

## 这份文件是干什么的

这份文件不是写给“懂机器学习的人”的。

它是专门写给“现在的你”的回顾文件，用来回答 5 个问题：

- 我们这篇论文到底在做什么
- 我问过什么
- 我们已经做了什么
- 为什么要这样做
- 我下一步应该看什么

以后每推进一步，我都应该继续更新这里。

## 先只记住一句话

这整个项目，本质上只是在做 3 件事：

1. 准备干净的数据
2. 训练不同模型做对比
3. 把结果整理成论文能写的结论

你现在之所以会觉得乱，不是因为你笨，而是因为项目已经从“写几段代码”变成了“完整实验系统”。

## 先把最容易卡住的词讲成人话

### `epoch`

白话：

- 模型把全部训练数据从头到尾学一遍，叫 `1` 个 `epoch`

你可以把它想成：

- 学生把整本练习册完整做一遍

为什么要有它：

- 训练 1 次通常不够
- 训练很多次，模型才会慢慢学会规律

为什么又不能无限训练：

- 训练太久，模型可能开始“死记硬背”

### `patience`

白话：

- 这是早停机制的“耐心值”

意思是：

- 如果模型在验证集上的表现连续很多轮都没有变好，那就别再浪费时间继续练了

你可以把它想成：

- 老师规定：如果你连续 20 次模考都没进步，就先停下来，别再盲目刷题

比如：

- `patience = 20`
- 表示连续 20 个 `epoch` 都没有进步，就停止训练

为什么这样做：

- 节省算力和时间
- 防止模型越练越“死板”

### `window_size`

白话：

- 模型每次要往前看多少天历史数据

比如：

- `window_size = 20`
- 意思是“用前 20 天的数据，预测第 21 天”

你可以把它想成：

- 你看最近 20 天的股票走势，再猜明天会怎样

为什么要比较 `20 / 60 / 100`：

- 看得太短，信息可能不够
- 看得太长，旧信息可能反而干扰判断

所以我们专门做“时间窗口敏感性分析”：

- 看不同窗口长度会不会影响模型表现

### `张量`

白话：

- 这是模型更喜欢的数据打包方式

普通 CSV 或 Excel 是二维表格。

但 `LSTM` / `GRU` 这类序列模型，需要的是三维结构：

- 第 1 维：一共有多少个样本
- 第 2 维：每个样本包含多少天
- 第 3 维：每天有多少个特征

所以张量不是“很高深的新数据”，而是：

- 把普通表格重新整理成模型能直接训练的格式

### `config`

白话：

- `config` 就是“这组实验到底是怎么设置的说明书”

里面会记：

- 用的是哪只股票
- 用的是 `LSTM` 还是 `GRU`
- `window_size` 是多少
- `hidden_size` 是多少
- `dropout` 是多少
- `learning_rate` 是多少

为什么你朋友会问这个：

- 因为论文实验讲究“可追溯”
- 你不能只有一个结果，却说不清它到底怎么跑出来的

你这个项目其实有两层 `config`：

- 每个实验子文件夹里的 `config.json`
- 顶层的 `configs/project_defaults.json`

这两个文件的区别是：

- `config.json`：记录某一组具体实验
- `project_defaults.json`：记录整个项目默认设定

### `dropout`

白话：

- 训练时故意让一部分神经元临时“休息”

为什么这样做：

- 防止模型把训练数据背死
- 让模型更稳一点

### `learning_rate`

白话：

- 模型每次改参数时，步子迈多大

你可以把它想成：

- 走路时每一步的步幅

太大：

- 容易一下跨过最优点

太小：

- 学得很慢

### `hidden_size`

白话：

- 模型内部“记忆盒子”的大小

小一点：

- 简单，但可能记不住足够信息

大一点：

- 能学更复杂的东西，但也更容易过拟合

### `Baseline` 和 `Proposed`

这两个词你后面会一直看到。

白话：

- `Baseline`：基础对照组，只用 `OHLCV`
- `Proposed`：增强组，在 `OHLCV` 基础上加技术指标

为什么要分两套：

- 因为论文不是只看“能不能预测”
- 还要看“加技术指标到底有没有帮助”

### `Early Stopping`

白话：

- 训练过程中自动判断“该不该停”

为什么要有它：

- 不让模型无意义地跑很多轮
- 也避免只因为训练时间更长而看起来更好

### `smoke test`

白话：

- 小范围冒烟测试

意思是：

- 先跑一小部分，看看代码通不通、会不会报错

为什么要有它：

- 正式跑大实验前先做安全检查

所以你看到的 `*_smoke*` 文件夹，不是正式论文结果，而是：

- 代码开发过程中的测试残留

## 你一直在问的“为什么这么做”，本质答案是什么

因为论文和普通写代码不一样。

普通写代码，很多时候“能跑”就行。

论文不行。论文必须回答：

- 为什么选这几只股票
- 为什么选 `LSTM` 和 `GRU`
- 为什么要加技术指标
- 为什么要比较不同 `window_size`
- 为什么这样切训练集和测试集
- 为什么这样设置超参数

所以我们每一步都不只是“把代码写出来”，而是在给论文后面的这三部分准备证据：

- Method
- Results
- Discussion

## 到目前为止，我们是怎么一步步做过来的

### 阶段 1：把题目定小

你一开始最困惑的是：

- 到底先做什么
- 题目怎么定
- 机器学习和股票都不熟怎么办

我们先做的不是写模型，而是把题目收窄。

为什么：

- 题目太大，后面一定失控

结果：

- 你的论文方向被定成了一个学生能完成的实证型对比研究

### 阶段 2：先拿到真实数据

我们下载了：

- `AAPL`
- `MSFT`
- `TSLA`

为什么不是一开始抓几十只股票：

- 先做小而干净的数据，更容易跑通流程
- 这三只股票公开、常见、文献里也常用

### 阶段 3：清洗数据并计算技术指标

我们做了：

- `RSI`
- `MACD`
- `Bollinger Bands`

同时你特别强调了学术红线：

- 技术指标产生的 NaN 必须直接 `dropna()`
- 不能用 `0`
- 不能用 `ffill` / `bfill`
- 必须先切分，再归一化

为什么这样做：

- 这是为了防止“看到了未来信息”
- 也是为了避免做出金融时间序列里不合理的填充

这一步可以简单理解成：

- 我们先保证数据干净，而且没有作弊

### 阶段 4：把普通表格变成模型能训练的数据

我们生成了：

- `window_size = 20`
- `window_size = 60`
- `window_size = 100`

每种窗口下又分成：

- `baseline`
- `proposed`

为什么：

- 模型不能直接吃表格
- 必须把表格切成“前 N 天预测下一天”的窗口样本

结果：

- 数据被保存成 `.pt`，后面训练可以直接读取

### 阶段 5：写单组训练引擎

我们写了 [train.py](C:/Users/27476/Desktop/论文/stock_prediction_paper/train.py)。

它负责：

- 读 `.pt` 数据
- 定义 `LSTM` / `GRU`
- 训练
- 验证
- 早停
- 评估
- 保存结果

为什么先写它：

- 因为所有批量实验，本质上都要靠它

你可以直接把它理解成：

- 这个项目的发动机

### 阶段 6：跑基础批量实验

我们写了 [run_all_experiments.py](C:/Users/27476/Desktop/论文/stock_prediction_paper/run_all_experiments.py)。

它做的是：

- 3 只股票
- 4 种模型/特征组合

总共 12 组。

为什么先跑这个：

- 先确认整体实验系统能稳定工作

### 阶段 7：跑终极窗口实验

后来你发现：

- `best_epoch = 100`
- 说明之前那版实验还不够充分

于是我们把：

- `max_epochs` 提到 `500`
- `patience` 提到 `20`
- 把 `window_size = 20 / 60 / 100` 都跑了一遍

这就是 [run_ultimate_experiments.py](C:/Users/27476/Desktop/论文/stock_prediction_paper/run_ultimate_experiments.py)。

它的作用是：

- 让实验更完整
- 真正支撑“时间窗口敏感性分析”

### 阶段 8：做网格搜索

你后面又觉得：

- 服务器算力还有余量
- 想把实验做得更扎实

所以我们写了 [run_grid_search.py](C:/Users/27476/Desktop/论文/stock_prediction_paper/run_grid_search.py)。

它会比较：

- `window_size`
- `hidden_size`
- `learning_rate`
- `dropout`

为什么这样做：

- 让论文不只是“随便挑了一组参数”
- 而是能更认真地说明：哪些超参数更合适

## 你现在看到那么多文件夹，到底怎么理解

你可以把它们先分成 4 类：

### 1. 数据区

- `data/`

### 2. 代码区

- `scripts/`
- `train.py`
- `run_ultimate_experiments.py`
- `run_grid_search.py`

### 3. 正式结果区

- `ultimate_training_runs/`
- `ultimate_results.csv`

### 4. 说明文档区

- `docs/`
- `configs/`

剩下那些看起来很乱的，多半属于：

- 早期结果
- 冒烟测试
- 缓存

它们不是最重要的论文主结果。
